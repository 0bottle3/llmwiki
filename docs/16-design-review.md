# 16. 설계 리뷰 & 결정 기록 (2026-06-01)

01~15번 설계 문서 전체를 감사하고, 발견한 모순·과설계·보안 리스크를
심각도별로 정리한 뒤 결정사항을 docs 전반에 반영한 기록.

## 목적 재확인

> **LLM이 팀의 도메인 지식·보고서·히스토리를 빠르게 파악해서,
> 사람이 매번 설명하지 않아도 요청을 잘 수행하게 한다.**

이 목적에서 진짜 가치는 **검색(MCP, 05번)** + **메모리 SSO(13번)** 두 곳에 있다.
인프라(EKS/Helm/네트워크)는 그 가치를 떠받치는 수단일 뿐 — 무게가 수단 쪽으로
쏠리지 않도록 MVP를 다이어트하는 것이 이번 리뷰의 큰 방향이었다.

---

## 확정된 결정

| ID | 결정 | 근거 |
|----|------|------|
| **C1 해결** | 가공은 **processor CronJob 단일화** (1시간 주기) | "S3 → 1시간 가공 → MCP" 라는 원래 구상과 일치. ingestor/SQS/DLQ/KEDA 제거 |
| **벡터 저장소** | **Qdrant on EKS** (파드 1개) | 추가비용 ~$0, 10명 규모에 충분. OpenSearch는 확장 경로로 강등 |
| D3 (기존) | 네트워크 게이트 + 호스트네임 식별 (인증서버 없음) | 유지 — 위협 모델 일관 |

OpenSearch Serverless / SQS+KEDA / Argo Rollouts는 **삭제가 아니라 "확장 경로
(Phase 2+)"** 로 강등. 요구가 생기면 다시 붙인다.

## 발견 사항 (심각도별)

### 🔴 Critical

**C1 — 문서 간 아키텍처 모순.**
01번은 "CronJob 단순화 / ingestor·SQS·KEDA 제거"라 선언했지만 03/06/07/12/15번은
옛 "SQS+KEDA+ingestor" 버전을 유지하고 있었다. 구현 시작 즉시 충돌하는 모순.
→ **CronJob 방향으로 전 문서 통일** (02·03·04·06·07·08·09·12·15 반영).

**C2 — 시크릿 마스킹 regex가 정상 문서를 죽인다.**
`r"[A-Za-z0-9+/=]{40,}"` 패턴이 코드 스니펫·git SHA·해시·base64 예제·JWT를
전부 [REDACTED] 처리하고, 게다가 검출 시 `SecretLeakError`로 **가공 실패→DLQ**.
가장 가치 있는 기술 문서가 조용히 검색에서 사라지는 폭탄.
→ 광범위 base64 패턴 제거, *접두사 명확한 형식 + 키워드 컨텍스트* 기반으로 교체.
검출돼도 **가공 실패가 아니라 "마스킹 후 통과 + `has_secrets` 플래그"** (07·15 반영).

**C3 — transcript 자동 전체공개의 프라이버시·시크릿 폭발.**
AI 대화를 통째로 S3에 올려 전원 검색 가능하게 하면 ① 디버깅 중 붙여넣은 prod
로그·PII·실토큰 유출 ② "남이 뭘 헤맸는지" 전부 열람 → 심리적 안전감 붕괴
③ 호스트네임 위장 시 사적 대화 오염. 12번은 슬랙엔 "명시 등록만" 원칙을 적용해
놓고, 정작 더 민감한 transcript엔 "다 빨아들임"이라 일관성이 깨져 있었다.
→ **opt-in + 기본 비공개(본인 prefix)**, 공유 표식(`share: team`)만 공용 인덱스.
search-api는 `sensitivity: private`를 **owner 본인에게만** 반환 (05·07·11 반영).

### 🟠 High (방향성)

- **H1 과설계** — 목표 대비 인프라가 2~3배. CronJob+Qdrant 결정으로 파드 5~6종
  → 3종(processor/search-api/curator)+업로드 게이트웨이로 축소. Argo Rollouts·
  OpenSearch·SQS·KEDA는 확장 경로로 강등.
- **H2 검증 순서** — 가장 큰 미검증 가설("가공 검색 품질이 쓸 만한가")을 3주차에야
  처음 확인하는 구조. → **Week 0(로컬 검증)** 권장 추가 (09 반영).
- **H3 latency** — Batch 24h + CronJob 1h = 최악 25h 지연. "방금 한 작업 이어서"엔
  너무 느림. → 대량 시드만 Batch, **증분(transcript 등)은 일반 API** 권장 (D6).

## 남은 권고 (선택 — 아직 미반영)

이번엔 "문서 정합성"에 집중했고, 아래는 **판단이 필요한 방향성 권고**라 남겨둔다:

1. **Week 0 로컬 dogfood를 실제로 먼저 할지** — EKS 착수 전 하루~이틀 로컬
   검증으로 검색 품질부터 확인하면 가장 큰 리스크를 가장 싸게 제거.
2. **D6를 "전부 Batch"에서 "시드만 Batch, 증분 realtime"으로 확정할지.**
3. **MVP에서 ingest-api(업로드 게이트웨이)도 뺄지** — 초기엔 본인 PC `aws s3 sync`
   로 시작하고, 팀 합류 시점에 zero-touch 게이트웨이를 붙이는 단계화도 가능.
